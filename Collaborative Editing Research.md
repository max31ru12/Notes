# Исследование: Совместное редакитрование

## Алгоритмы совместного редактирования:

- Operational Transformations (OT)
- Conflict-free Replicated Data Types (CRDT)

Также есть другие подходы, например, `блокирование документа`, `блокирование части документа` и `системы контроля версий`. Эти варианты нам не подходят, потому что они либо не **user friendly** (контроль версий), либо выполняют другие задачи (остальные).

Также есть интересный вариант с использованием подходов `OT` и `differential transformations`. Готовых решений и какой-то инфы, скорее всего, нет. 

## Operational Tranformations

OT - это про <ins>**consistensy**</ins> и преобразование операций с учетом других операций. 

Невероятно крутые статьи (2 части) на понимание процессов ОТ и интересный подход с ревизиями для состояний документов:

- https://habr.com/ru/companies/ncloudtech/articles/264923/
- https://habr.com/ru/companies/ncloudtech/articles/266429/

OT часто используется для редактирования текстовых документов.
Особенности: 
- Скорее всего придется реализовывать операции и преобразования самостоятельно
- Активно используется центральный сервер (бэк)
- Не так сильно привязан к структурам данных и их сложности, потому что можно самостоятельно все разработать 
- Масштабирование??? `Сложно`, но можно. У CRDT лучше.

Из недостатков:
- Сложность алгоритмов
- Нагрузка на сеть и сервер

Из преимуществ:
- Согласованность БД 
- Разрешение конфликтов 
- Отлично решение для редактирования доков

Задача сложная, но интересная. Однозначно можно использовать для задачи редактирования документов. Вряд ли подойдет для интерактивной доски.


## Conflict-free Replicated Data Types

Это про <ins>**Eventual consistensy**</ins>. Для мгновенного отклика на фронте базовую логику CRDT и преобразования локального состояния лучше реализовать на фронте. Бэк - синхронизация локального и глобального состояний, разрешение конфликтов, хранение глобального состояния.

1. Локальное состояние - на клиенте
2. Глобальное состояние  - на сервере

Особенности:
- Не такая сильная привязка к серверу, как у OT
- Локальное и глобальное состояния в один момент времени могут различаться
- Большая часть операционных преобразований происходит за счет `conflict-free` структур данных ([примерное описание этих самых структур](https://www.baeldung.com/java-conflict-free-replicated-data-types))
- Сервер хранит глобальное состояние
- Сервер получает локальное состояние или отдельные операции и сливает с глобальным состоянием 
- Сервер рассылает глобальное состояние всем клиентам

Недостатки:
- Сложность алгоритма зависит от сложности структур данных

Преимущества:
- Изменения имеют моментальный отклик для клиента
- Есть готовые либы (Yjs уже использует CRDT)
- Меньше нагрузки на сервер  

Эта задача как раз подойдет для интерактивной доски. 

Фронтендерские либы:

- [AFFiNE Docs](https://docs.affine.pro/docs/hello-bonjour-aloha-你好) - используется CRDT

- [Excalidraw](https://docs.excalidraw.com/) - непонятно, какой подход используется

- [Slate](https://docs.slatejs.org/) - выглядит здорово, дока вроде более менее, скорее всего тоже использует CRDT (для совместного редактирования либа требует [Yjs-провайдер](https://docs.slatejs.org/walkthroughs/07-enabling-collaborative-editing))

## Как будем реализовывать совместное редактирование

**Редактор документов** - использовать ОТ вместе с этой [штукой](https://www.geeksforgeeks.org/real-time-collaborative-editing-app-using-react-websockets/?ref=ml_lbp). Фронту мб придется делать больше работы, но зато с бэком все куда понятнее: разрабатываем только те операции, которые нужны, без костылизирования под готовые либы. Стоит попробовать подход с ревизиями из статей на хабре, которые приведены выше, выглядит перпективно.

**Доска** - костылизируем свои решения под готовые либы. Работаем с теми структурами данных, которые нам предложит фронт.
